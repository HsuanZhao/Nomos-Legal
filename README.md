## 🎯 核心理念

**法学方法论 × 编程语言 = 可验算的裁判技术**

Nomos-Legal 是一门**法律领域特定语言（Legal Domain-Specific Language）**，它将民法教义学中的两大核心工具——**涵摄**与**请求权基础**——转化为计算机可判定的形式化逻辑框架。

### 🔍 涵摄 (Subsumption)

涵摄是法律适用的核心技术——将具体案件事实置入法律规范的构成要件之下，以得出法律效果的逻辑三段论。

在 Nomos-Legal 中，涵摄被编码为 **"要件 → 事实 → 结论"** 的强制映射结构：

| 三段论要素 | Nomos-Legal 组件 | 涵摄角色 |
| :--- | :--- | :--- |
| **大前提** | `spec` (规范) | 法律规范的构成要件与法律效果 |
| **小前提** | `impl` (事实) | 具体的、被证据证明的案件事实 |
| **结论** | `Verifier` (推理引擎) | 判断事实是否符合要件的逻辑归纳 |

### 📜 请求权基础 (Anspruchsgrundlage)

请求权基础是诉讼结构的核心命题——**"谁得向谁，依据何种规范，主张何种权利"**。

在 Nomos-Legal 中，`claim` 关键字强制用户在编码开始时即明确：

nomos
claim "借款返还请求权" {
    basis = "民法典第667条";   // 依据何种规范
    plaintiff = "出借人";       // 谁
    defendant = "借款人";       // 向谁
    relief = "返还本金及利息";  // 主张何种权利
}

## 🤔 为什么需要 Nomos-Legal？

### 法律实务中的痛点

| 痛点 | 传统方式 | Nomos-Legal 的解法 |
| :--- | :--- | :--- |
| **法条解释存在歧义** | 依赖个人理解，缺乏统一标准 | 将法条拆解为原子化逻辑谓词，消除歧义 |
| **裁判推理难以追溯** | 判决书仅展示结论，推理过程不透明 | 生成完整的"涵摄树（Proof Tree）"，每一步都可追溯 |
| **数学计算易出错** | 利息、违约金、赔偿金等依赖人工计算 | 内置数学约束引擎，自动验算并强制合规 |
| **请求权检索遗漏** | 可能遗漏备位请求权 | 自动检索所有关联请求权，按优先级排序 |
| **新人学习曲线陡峭** | 依赖师徒制口传心授 | 将方法论编码为结构化语法，可系统化学习 |

### 谁适合使用 Nomos-Legal？

- **法律科技开发者**：构建智能法律检索、智能裁判辅助系统
- **执业律师**：案件预判、诉讼策略模拟、法律文书生成
- **法学研究者**：验证法律推理的逻辑一致性、进行实证研究
- **法学教育者与学生**：系统学习请求权基础分析法与涵摄技术

## ✨ 特性

### 一、请求权规范竞合

支持在同一案件中同时检索多个请求权基础，并按优先级自动判断：
'''
nomos
claim "合同违约请求权" {
    basis = "民法典第577条";
    plaintiff = 买方;
    defendant = 卖方;
    relief = 损害赔偿;
    priority = "primary";   // 优先审查
}

claim "侵权损害赔偿请求权" {
    basis = "民法典第1165条";
    plaintiff = 买方;
    defendant = 卖方;
    relief = 侵权赔偿;
    priority = "secondary"; // 备位审查
}


### 二、要件事实原子化

将《民法典》中的法律规范拆解为最小逻辑单元：

nomos
spec "民法典第143条_民事法律行为有效要件" {
    element 有效要件 {
        要件一 = 行为人具有相应的民事行为能力;
        要件二 = 意思表示真实;
        要件三 = 不违反法律、行政法规的强制性规定;
        要件四 = 不违背公序良俗;
    }
    // 四要件必须全部满足（AND关系）
    rule 判断有效 {
        if (要件一 AND 要件二 AND 要件三 AND 要件四) {
            法律效果 = "民事法律行为有效";
        }
    }
}

### 三、数学约束引擎

将法律中的"上限"、"限额"、"调减"规则编码为可验算的数学约束：

| 法律规则 | 数学约束 | 对应法条 |
| :--- | :--- | :--- |
| 利率上限 | `利息 ≤ 本金 × LPR×4 × 期限` | 民法典第680条 |
| 砍头息扣除 | `实际本金 = 合同金额 - 预扣利息` | 民法典第670条 |
| 违约金调减 | `违约金 ≤ 实际损失 × 1.3` | 民法典第585条 + 司法解释 |
| 复利限制 | `本息和 ≤ 本金 × (1 + LPR×4 × 总期限)` | 民间借贷司法解释第27条 |

### 四、涵摄树可视化

Verifier 输出包含完整涵摄过程的 JSON 报告：

json
{
  "subsumption_tree": {
    "大前提_构成要件_借贷合意": {
      "规范要求": "存在借据或合同或欠条",
      "案件事实": "存在借据_证据 (signed_by=被告)",
      "涵摄结论": "符合 ✓"
    },
    "大前提_构成要件_款项交付": {
      "规范要求": "存在资金交付记录",
      "案件事实": "存在转账记录 (amount=1000000)",
      "涵摄结论": "符合 ✓"
    }
  }
}

### 五、举证责任动态转移

自动实现《民事诉讼法》中的举证责任分配与转移机制：

- 原告初步举证 → 举证责任转移至被告
- 被告抗辩但无证据 → 抗辩不成立
- 被告提供反证 → 举证责任回归原告


## 🏛️ 架构总览

Nomos-Legal 采用 **"请求权锚定 → 规范要件 → 事实涵摄 → 效果计算"** 的四层架构：
┌─────────────────────────────────────────────────────────────────┐
│ Nomos-Legal 四层涵摄模型 │
├─────────────────────────────────────────────────────────────────┤
│ │
│ 第一层：请求权锚定 (Claim) ← 定性：谁，向谁，依何法 │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ claim "借款返还请求权" { basis, plaintiff, defendant } │ │
│ └─────────────────────────────────────────────────────────┘ │
│ ↓ │
│ 第二层：规范大前提 (Spec IR) ← 大前提：构成要件 + 效果 │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ spec "民法典第667条" { element 合意; element 交付; } │ │
│ └─────────────────────────────────────────────────────────┘ │
│ ↓ │
│ 第三层：事实小前提 (Impl IR) ← 小前提：证据 + 事实 │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ impl "案件事实" { evidence 借据; evidence 转账记录; } │ │
│ └─────────────────────────────────────────────────────────┘ │
│ ↓ │
│ 第四层：法律效果量化 (Target) ← 结论：数学计算 + 约束 │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ target "偿还额" { maximize; constraints {利率≤LPR×4} } │ │
│ └─────────────────────────────────────────────────────────┘ │
│ ↓ │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ Verifier 涵摄推理引擎 │ │
│ │ 输入：Spec IR + Impl IR + Platform Config │ │
│ │ 输出：涵摄树 + 裁决报告 + 数学结果 │ │
│ └─────────────────────────────────────────────────────────┘ │
│ │
└─────────────────────────────────────────────────────────────────┘


### 五层模型详解

| 层级 | 关键字 | 功能 | 类比编程 |
| :--- | :--- | :--- | :--- |
| **平台配置** | `platform` | 注入外部变量（LPR、诉讼时效、地域标准） | 环境变量 / ConfigMap |
| **请求权锚定** | `claim` | 声明请求权基础（谁→谁→依何法→主张何权） | API 端点定义 |
| **规范大前提** | `spec` | 定义法律规范的构成要件与法律效果 | 接口 / 业务规则 |
| **事实小前提** | `impl` | 绑定具体案件证据与事实 | 实例化对象 / 入参 |
| **效果量化** | `target` | 将法律效果转化为数学最优化问题 | 目标函数 / Solver |


## 🚀 快速开始

### 安装

bash
# 克隆仓库
git clone https://github.com/your-org/nomos-legal.git
cd nomos-legal

# 创建虚拟环境（推荐）
python -m venv venv

# 安装依赖
pip install -r requirements.txt

# 设置命令行别名（可选）
alias nomos="python -m nomos.cli"

### 第一个程序
创建一个 demo.legal 文件：


@legal "民间借贷纠纷 - 简易示范"
version "1.0.0"

// 1. 平台配置
platform {
    lpr_year = 3.45%;
    lpr_four_times = 13.8%;
}

// 2. 目标（法律效果量化）
target "借款偿还额" {
    let 应还本金 = 100000;
    let 法定利率上限 = platform.lpr_four_times;
    let 借款年数 = 1;
    
    maximize claimable = 应还本金 + 合法利息;
    constraints {
        合法利息 <= 应还本金 * 法定利率上限 * 借款年数;
    }
}

// 3. 规范大前提（涵摄基准）
spec "民法典第667条_借款返还请求权" {
    element 成立要件 {
        借贷合意 = exists(借据) OR exists(借款合同);
        款项交付 = exists(转账记录) OR exists(现金交付凭证);
    }
    
    rule 涵摄_成立判断 {
        if (成立要件.借贷合意 AND 成立要件.款项交付) {
            请求权状态 = "成立";
        } else {
            请求权状态 = "不成立";
        }
    }
}

// 4. 事实小前提
impl "张三诉李四" {
    entity 原告 = "张三";
    entity 被告 = "李四";
    
    evidence 借据 {
        exists = true;
        signed_by = 被告;
        amount = 100000;
    }
    
    evidence 转账记录 {
        exists = true;
        amount = 100000;
        from = 原告;
        to = 被告;
    }
    
    contract 实际借款金额 = 100000;
    contract 约定利率 = 20%;
    contract 借款年数 = 1;
}

// 5. 请求权声明
claim "张三_借款返还请求权" {
    basis = "民法典第667条";
    plaintiff = impl.原告;
    defendant = impl.被告;
    relief = target.借款偿还额;
}


### 运行
'''
nomos run demo.legal --output report.json --verbose
'''
